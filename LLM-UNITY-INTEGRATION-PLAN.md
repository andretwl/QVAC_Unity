## Phase 2: QVAC ↔ LLMUnity Integration Architecture

**Goal**: Decide the integration mode and set up the connection.

### Three Integration Options

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A. Remote API (Recommended)** | Unity LLMClient connects to QVAC's remote server | Shared models, centralized LoRAs, hot-swappable | LLMUnity remote protocol uses llama.cpp native (not OpenAI API) — may need bridge |
| **B. Local GGUF + LoRA** | Unity loads GGUF directly + LoRA adapters from QVAC | No network dependency, runs offline | Each NPC needs model reload for different LoRAs, no centralized management |
| **C. Hybrid** | Base model local, LoRAs applied at load time | Best of both if multiple characters share base | More setup complexity |

### Recommended Approach (Hybrid)
1. **QVAC serves the primary model** (infrequent reloads)
2. **Unity loads LoRA adapters** generated by QVAC's `finetune()` for character-specific voices
3. **QVAC RAG** provides NPC knowledge base (lore, quest info, backstory)
4. **QVAC embeddings** enhance Unity's semantic search for dialogue context

### Key Integration Points
- **LoRA Format**: GGUF-based LoRA adapters (`.gguf`) — QVAC `finetune()` outputs adapter parameters, Unity `LLM.AddLora(path, weight)` consumes them
- **Model Sharing**: Both use llama.cpp GGUF format; point Unity to the same model cache at `/mnt/data/projects/QVAC/.qvac/models/`
- **RAG**: QVAC's RAG API can pre-compute embeddings → Unity's built-in RAG system consumes them for NPC knowledge

## Phase 3: NPC LoRA Fine-Tuning Pipeline (via QVAC)

**Goal**: Create a repeatable pipeline to train personality-specific LoRA adapters for each NPC.

### QVAC Finetune API (already available in SDK v0.13.5)
```
import { finetune } from "@qvac/sdk";

const handle = finetune({
  modelId,
  options: {
    trainDatasetDir: "./datasets/npc_merchant/",
    validation: { type: "split", fraction: 0.05 },
    outputParametersDir: "./artifacts/lora/npc_merchant/",
    numberOfEpochs: 3,
    learningRate: 1e-4,
    loraRank: 16,
    loraAlpha: 32,
    loraModules: "q_proj,v_proj",  // target attention modules
    checkpointSaveSteps: 50,
  },
});
for await (const progress of handle.progressStream) {
  console.log(`Step ${progress.global_steps}: loss=${progress.loss}`);
}
const result = await handle.result;
```

### Dataset Format
Each NPC needs a training dataset directory with this structure:
```
datasets/npc_merchant/
├── train/
│   ├── conversation_01.jsonl   # {"messages": [{"role": "user", "content": "...", "role": "assistant", "content": "..."}]}
│   ├── conversation_02.jsonl
│   └── ...
├── test/                        # optional holdout (if using split, omit this)
│   └── ...
```

### NPC Dialogue Dataset Design
Each NPC conversation should demonstrate:
1. **Character voice** — colloquialisms, speech patterns, vocabulary
2. **Knowledge domain** — what this NPC knows (lore, shop items, quest info)
3. **Behavioral traits** — personality (grumpy, helpful, mysterious, comedic)
4. **Response patterns** — how they react to common situations

### Example NPCs
| NPC | Personality | Dataset Size Target | LoRA Rank |
|-----|-------------|-------------------|-----------|
| Merchant | Chatty, bartering-focused | 50-100 conversations | 16 |
| Quest Giver | Formal, lore-rich | 30-80 conversations | 16 |
| Tavern Keeper | Warm, rumor-spreading | 40-80 conversations | 8 |
| Guard | Terse, duty-focused | 20-50 conversations | 8 |

## Phase 4: Unity Implementation

**Goal**: Wire up NPCs in Unity with LoRA-enhanced dialogue.

### In Unity Editor (via GladeKit or manually)
1. **Create LLM GameObject** — point to your base GGUF model
   ```
   LLM.model = "Qwen3-1.7B-Q4_0.gguf"
   LLM.numGPULayers = 99
   LLM.contextSize = 4096
   ```

2. **Attach LoRA adapters per NPC** — use `AddLora()` programmatically:
   ```csharp
   // For a unique NPC voice
   llm.AddLora("npc_merchant_lora.gguf", 0.8f);
   // Future: swap LoRA for different NPC conversations
   llm.RemoveLora("npc_merchant_lora.gguf");
   llm.AddLora("npc_guard_lora.gguf", 0.8f);
   ```

3. **Configure LLMAgent per NPC**:
   ```csharp
   LLMAgent npcAgent = gameObject.AddComponent<LLMAgent>();
   npcAgent.llm = llm;  // shared LLM instance
   npcAgent.systemPrompt = "You are a grizzled dwarf merchant...";
   npcAgent.temperature = 0.7f;
   npcAgent.topP = 0.9f;
   npcAgent.numPredict = 150;
   npcAgent.save = "npc_merchant_history.json";
   ```

4. **Optional: RAG for NPC Knowledge**:
   ```csharp
   RAG npcRag = npcGO.AddComponent<RAG>();
   npcRag.Init(SearchMethods.DBSearch, ChunkingMethods.SentenceSplitter, llm);
   foreach (string loreEntry in npcLore) await npcRag.Add(loreEntry, "merchant");
   
   // In Chat handler:
   var (results, distances) = await npcRag.Search(playerQuery, 3, "merchant");
   string enrichedPrompt = $"Based on this data: {string.Join(", ", results)}\nAnswer:";
   ```

## Phase 5: Dialogue UI & Interaction Flow

**Goal**: Build the player ↔ NPC dialogue system.

### Architecture
```
Player Input → Dialogue Manager → LLMAgent.Chat() → LLM (with LoRA) → Response
                                   ↓
                              RAG Search (knowledge base)
                                   ↓
                            Response → UI (Chat bubbles or NPC speech)
```

### Implementation Options (from existing samples)
- **ChatBot style**: Bubble-chat UI (already in `Assets/LLMUnity/Samples/ChatBot/`)
- **MultipleCharacters**: Dropdown to switch NPCs (already in `Assets/LLMUnity/Samples/MultipleCharacters/`)
- **FunctionCalling**: Structured actions from NPC (already in `Assets/LLMUnity/Samples/FunctionCalling/`)
- **Custom**: Create a "talk to NPC" interaction system

### Recommended Flow
1. Player approaches NPC and presses interaction key
2. Dialogue UI opens with NPC portrait and name
3. Player types/selects dialogue option
4. `LLMAgent.Chat()` sends to LLM with NPC's loaded LoRA
5. Response streams back into UI bubble
6. Player can respond or end conversation

## Notes & Discoveries

### LLMUnity Remote Protocol
LLMUnity's remote mode uses a **native llama.cpp protocol** (via `LLMClient_Construct_Remote`), not the standard OpenAI API. This means direct HTTP integration requires either:
- A translation bridge/proxy
- Or using local mode with shared GGUF files + LoRAs

The **hybrid approach** (local GGUF + QVAC-trained LoRAs) avoids protocol mismatch entirely.

### QVAC Finetune Output
QVAC's `finetune()` writes LoRA adapter parameters to a directory (not a single `.gguf` file). We may need to:
- Convert the output to a `.gguf` LoRA format compatible with llama.cpp
- Or use QVAC's own `loadModel` with the LoRA path to test in-server before exporting

### Model Compatibility
LLMUnity uses the same underlying llama.cpp library. Any GGUF model QVAC can load, LLMUnity can load too (given sufficient VRAM). The cached models at `.qvac/models/` can be shared.
